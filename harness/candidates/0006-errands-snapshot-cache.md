# Candidate 0006: cache/truncate the Errands-tab colony-wide chore list rebuild

- Status: PARKED (declined — not worth building; 0005 already took the dominant win, see Recommendation)
- Target: `MinionTodoSideScreen.PopulateElements` (`harness/decompiled/Assembly-CSharp/MinionTodoSideScreen.cs:156-231`), reading `ChoreConsumer.GetLastPreconditionSnapshot()` (`harness/decompiled/Assembly-CSharp/ChoreConsumer.cs:135-138`) and, for approach C, a new patch on `ChoreConsumer.FindNextChore` (`ChoreConsumer.cs:303-329`)
- Risk class: Approach C = new Postfix on a colony-wide sim-hot method (`FindNextChore`) + Prefix rework on `PopulateElements`. Approach B = Transpiler (surgical loop cap) or full Prefix replacement (not recommended) on `PopulateElements` only.
- Gating flag: `SideScreenOpts` (same flag as 0005, for either approach)
- Collision check: grepped `FastTrack/**/*.cs` for `ChoreConsumer`, `PreconditionSnapshot`, `FindNextChore` — the only hits are `FastTrack/UIPatches/MinionTodoSideScreenPatches.cs` (0005's throttle) and the `harmony.Profile(typeof(MinionTodoSideScreen), "PopulateElements")` line in `FastTrack/FastTrackMod.cs`. Nothing in the fork touches `ChoreConsumer`, `FindNextChore`, or `PreconditionSnapshot` today. Clean, no overlap.

## What and why

0005 cut `PopulateElements` call frequency ~6-7x (60-70/sec -> ~8-10/sec) by throttling the Errands-tab rebuild to the panel's own 0.1s cadence. Each surviving call still costs ~2-8ms because it fully re-derives its render list from the sim's `PreconditionSnapshot` every time, even though the sim only actually changes that snapshot on a brain tick. This candidate investigates whether that remaining per-call cost can be cut further, either by (C) skipping the rebuild entirely when the snapshot hasn't changed since the last render, or (B) capping the rebuild to the top N highest-priority entries.

## Investigation findings

**1. No existing O(1) change-signal.** `ChoreConsumer` and `PreconditionSnapshot` (`ChoreConsumer.cs:20-42`) expose no version counter, timestamp, tick number, or changed-event. The only stateful flag is `PreconditionSnapshot.doFailedContextsNeedSorting` (`ChoreConsumer.cs:26`), but that means "failedContexts hasn't been sorted since the last `Clear()`," not "content changed since last render" — and `PopulateElements` itself already consumes and resets that flag for its own sort-memoization (`MinionTodoSideScreen.cs:162-166`). There is nothing else to read. **No exact `Type.member` exists for this today.**

**2. Snapshot is mutated in place, not replaced.** `ChoreConsumer.preconditionSnapshot` (`ChoreConsumer.cs:95`) is constructed once as a field initializer and never reassigned. `FindNextChore` calls `preconditionSnapshot.Clear()` then repopulates the same `List<Chore.Precondition.Context>` instances in place (`ChoreConsumer.cs:305-321`). `GetLastPreconditionSnapshot()` (`ChoreConsumer.cs:135-138`) always returns the same object reference. **This means a naive "cache the reference, skip if `==`" check is worthless — the reference never changes even when content does.** Any real change-detection needs either a new counter bumped by `FindNextChore` (doesn't exist, would have to be added) or content hashing (which the prompt correctly identifies as defeating the point at colony scale).

**3. `FindNextChore`'s only callers are `Brain.FindBetterChore`→`Brain.UpdateChores`→`Brain.UpdateBrain`** (`Brain.cs:44-67`) **and `SolidTransferArm` (a separate consumer path, not relevant to dupes)**. `Brain.UpdateBrain` is invoked from `BrainScheduler.DupeBrainGroup.RenderEveryTick` (`BrainScheduler.cs:90-119`), a round-robin, load-balanced probe across every dupe brain in the colony each sim tick, plus a `priorityBrains` queue that lets specific brains (e.g. one whose chore just ended) jump the queue for near-immediate re-evaluation. There is **no fixed "every few seconds" timer** — cadence depends on colony size, load-balance tuning, and event-driven priority re-checks. Critically: this scheduler runs for **every dupe in the colony**, not just the one with an Errands tab open. Any counter added here to support approach C would execute on a hot, colony-wide, always-running path purely to benefit an occasionally-open single UI panel — a bad cost/benefit ratio unless the increment is genuinely free (see Patch section below for the mechanism and its cost).

**4. The 2-8ms is UI-side, not a re-payment of sim cost.** `PopulateElements` never calls `FindNextChore` or `GlobalChoreProvider.CollectChores` — it only reads the two already-built lists off the snapshot. Its own cost is real re-derivation work performed every single call regardless of whether the snapshot changed: two `pooledList.AddRange(...)` copies of the full `failedContexts`/`succeededContexts` lists (`MinionTodoSideScreen.cs:167-168`, O(N) colony/backlog-scaled), a full backward iteration testing `chore != null` / `target.isNull` / `IsPotentialSuccess()` / `GameUtil.AreChoresUIMergeable` per surviving entry, and a linear `PriorityGroupForPriority` scan (up to ~28 groups) per entry that isn't merged. **Confirmed split: sim-side `CollectChores`/precondition-testing cost is amortized on the brain-tick cadence and is NOT re-paid inside `PopulateElements`; the ~2-8ms is entirely `PopulateElements`'s own list-copy/filter/widget-sync loop, re-run from scratch every call.** This is exactly the waste 0006 was proposed to address — confirming the premise, not just speculating on it.

**5. Per-widget updates are already partially memoized, which changes the staleness-risk picture.** `MinionTodoChoreEntry.Apply` (`MinionTodoChoreEntry.cs:47-53`) already early-returns when `targetChore == context.chore && target == lastChoreTarget && masterPriority == lastPrioritySetting` — so text/sprite/tooltip recompute is already skipped for unchanged entries today. There is no animation, live countdown, or per-frame visual state in `MinionTodoChoreEntry` (it's static text + icon + tooltip; confirmed by full read of the file) — so a "skip the whole rebuild when truly unchanged" cache would not introduce any visible animation/countdown staleness bug. However, `MinionTodoSideScreen.GetChoreEntry` (`MinionTodoSideScreen.cs:233-250`) unconditionally calls `transform.SetParent(...)`, `SetAsLastSibling()`, and `gameObject.SetActive(true)` on every surviving entry every call — real Unity transform/hierarchy work that runs even though `Apply`'s own guard is a no-op on unchanged data. That reparent/reactivate churn, plus the list copy/filter pass, is the concrete waste a "skip whole rebuild" cache would remove. The residual risk of skipping is bounded to "chore reassignment/priority change lags by up to one brain tick," which is the same class of staleness 0005's review already accepted (≤100ms dupe-switch lag) — not a new risk category, just a slightly larger and less bounded window (a brain tick, not a fixed 100ms).

## Approach B assessment (truncate to top N)

`PopulateElements` builds `pooledList` as `failedContexts` (lazily sorted) followed by `succeededContexts` (sorted ascending in `FindNextChore`, `ChoreConsumer.cs:321`), then iterates `pooledList` **backward** from `Count-1` to `0` (`MinionTodoSideScreen.cs:185`) — the same backward-from-highest-index pattern `ChoreConsumer.ChooseChore` uses over `succeededContexts` (`ChoreConsumer.cs:276, 290`), which strongly implies ascending sort with highest priority at the tail, i.e. the loop visits the current/highest-priority chores first and the low-priority "basic"/"idle" tail last. This is favorable for truncation: capping "stop after N new widgets created" would naturally preserve the current task and top-priority entries and cut exactly the long tail of low-priority backlog entries that are the pathological case motivating this whole candidate.

Invasiveness: capping cannot be a Postfix (the loop has already fully run by the time a Postfix sees it) and cannot be a naive Prefix skip (unlike 0005, this needs to change behavior partway through the method, not skip it wholesale). Two options:
- **Transpiler**: inject a counter + early-exit check into the existing backward `for` loop's IL, only around the widget-creation branch (`MinionTodoSideScreen.cs:210-217`), leaving the `currentTask` detection (line 189) and merge detection (line 197) untouched so those special cases are never accidentally truncated. Moderate invasiveness, surgical, but IL loop-splicing has real correctness risk (matching the exact loop/branch structure, staying correct across game updates) and doesn't reduce the `AddRange` copy or the O(N) filter conditions — those still run over the full list every call. So B only removes the tail-end widget instantiate/reparent cost, not the list-copy/filter cost, meaning its win is a fraction of the remaining 2-8ms, not most of it.
- **Full Prefix replacement**: reimplement the whole ~80-line grouping/pooling/merge method with a cap baked in. Higher invasiveness (duplicated logic that silently drifts from Klei's original on future game updates), not recommended.

UX cost: acceptable in principle (truncates low-priority backlog entries a player is least likely to be checking), but needs a "N of M hidden" indicator to avoid silently hiding entries, which doesn't exist today and adds scope. Sort-direction assumption is inferred from `ChooseChore`'s matching backward-iteration convention, not independently confirmed by reading `Chore.Precondition.Context.CompareTo` — medium, not full, confidence.

## Predicted impact

- Approach C: if built, would eliminate close to all of the remaining ~2-8ms/call *only when the brain hasn't ticked between renders* — but the mechanism to detect that requires instrumenting `ChoreConsumer.FindNextChore`, which runs continuously for every dupe in the colony regardless of whether any Errands tab is open. The benefit is scoped to a single open UI panel; the cost of the new instrumentation is colony-wide and constant. Net impact is uncertain and could be negative on large colonies if the per-instance bookkeeping isn't truly free.
- Approach B: removes the tail-end widget instantiate/reparent cost (bounded portion of the per-call cost) on backlogged colonies with long tails of low-priority entries; does not reduce the list-copy/filter cost, so the win is partial, not most of the remaining 2-8ms.
- Baseline: 0005 already reduced this path from ~240ms/sec to ~32ms/sec while a single dupe's Errands tab is open. Both C and B chase a bounded, single-panel residual cost, not a colony-wide one.

## Patch (design-level only, not built)

**If C were built:** add `[HarmonyPatch(typeof(ChoreConsumer), "FindNextChore")]` Postfix that bumps a generation counter for `__instance` (game code has no field to reuse; would need an external keyed store, e.g. `ConditionalWeakTable<ChoreConsumer, StrongBox<int>>` — a pattern used exactly once elsewhere in this fork, `VisualPatches/KAnimBatchPatches.cs`, so precedented but not idiomatic here). `MinionTodoSideScreenPatches` would track `(ChoreConsumer, int generation)` last seen (parallel to 0005's existing static `lastPopulate`), and skip the body when generation is unchanged. Because `PopulateElements` re-arms its own 0.1s scheduler handle at the top of its body (the mechanism 0005's review carefully reasoned through), a generation-based skip would also skip that re-arm — meaning the Prefix would need to replicate the `ClearScheduler`/`Schedule` bookkeeping itself on the skip path, not just return `false`. That's extra plumbing layered onto an already load-bearing throttle, raising the risk of breaking 0005's self-sustaining loop.

**If B were built:** a Transpiler on `PopulateElements` inserting a created-count check around the widget-creation branch only (line ~211), capped at a tunable N (e.g. 30-50), gated by the same `SideScreenOpts` flag, leaving `currentTask`/merge branches unmodified.

## Outcome

**Recommend NEITHER as a primary build right now.**

- **Approach C is not recommended.** No O(1) change-signal exists in `ChoreConsumer`/`PreconditionSnapshot` today (confirmed by full read — no counter, timestamp, tick, or event). Building one requires patching `ChoreConsumer.FindNextChore`, which is a colony-wide sim-hot path (every dupe, every brain-scheduler probe) with no fixed cadence — a much hotter, more central method than the single-panel UI method it would optimize. That inversion (touch a hot path used by the whole colony's AI, to save cost on an occasionally-open UI panel) is a bad risk/benefit trade, and it also requires reworking 0005's carefully-reasoned self-rescheduling mechanism to still re-arm on the skip path. The mechanism is buildable but not clean, and not worth the new hot-path surface.
- **Approach B is feasible but only a partial win.** The sort-order evidence (backward iteration matching `ChooseChore`'s convention) supports truncating the low-priority tail without losing the current task, and it stays UI-only (no sim-hot-path patch). But it only removes the tail widget-instantiate/reparent cost — the `AddRange` copy and O(N) filter pass over the full list still run every call — so it doesn't close most of the remaining 2-8ms, and doing it safely needs an IL transpiler (correctness risk around `currentTask`/merge branch preservation) rather than a simple loop-count cap.
- **Bottom line:** 0005 already delivered the dominant win here (6-7x fewer rebuilds, ~240ms/sec -> ~32ms/sec) and the remaining per-call cost is bounded to whichever single dupe currently has its Errands tab open on a backlogged colony — not a colony-wide, continuous cost. Given that scope, neither the sim-hot-path risk of C nor the partial, transpiler-risk win of B clears the bar today. If future profiling shows `PopulateElements` still surfacing as a real frame-time spike during normal play (not just a synthetic backlog test), B is the fallback worth revisiting first — it's UI-scoped and its mechanism is understood; C should not be pursued unless Klei's own code later adds a real snapshot-generation signal to `ChoreConsumer` that a patch could read for free.
